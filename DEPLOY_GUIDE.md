Guia completo para deploy do projeto fora do Figma Make.

## 3. Deploy do Frontend

### 3.1. Baixar o projeto

No Figma Make, clique no menu → "Download" para baixar o ZIP com todos os arquivos.

### 3.2. Ajustar o arquivo de conexao Supabase

O arquivo `/utils/supabase/info.tsx` contem o project ID e anon key hardcoded.
Se voce mudar de projeto Supabase, atualize esses valores:

```tsx
// /utils/supabase/info.tsx
export const projectId = "SEU_PROJECT_ID"
export const publicAnonKey = "SUA_ANON_KEY"
```

### 3.3. Ajustar o `package.json`

Adicione o script `dev` e mova `react`/`react-dom` de peer para dependencies:

```json
{
  "scripts": {
    "dev": "vite",
    "build": "vite build",
    "preview": "vite preview"
  },
  "dependencies": {
    "react": "18.3.1",
    "react-dom": "18.3.1"
  }
}
```

> **IMPORTANTE**: No Figma Make, `react` e `react-dom` sao `peerDependencies`
> porque a plataforma os fornece automaticamente. No seu servidor, eles
> precisam estar em `dependencies`.

### 3.4. Instalar dependencias e buildar

```bash
cd carretao-auto-pecas/

# Instalar dependencias
npm install

# Build de producao
npm run build

# Resultado: pasta /dist com os arquivos estaticos
```

### 3.5. Verificar o build

```bash
# Servir localmente para testar
npm run preview
# Abrir http://localhost:4173

# Abrir analise de chunks (gerado pelo rollup-plugin-visualizer)
open stats.html
```

### 3.6. Opcao A — Deploy no Vercel (recomendado, gratis)

```bash
# Instalar Vercel CLI
npm install -g vercel

# Na raiz do projeto:
vercel

# Configuracoes:
#   Framework: Vite
#   Build Command: vite build
#   Output Directory: dist
```

Criar arquivo `vercel.json` na raiz para SPA routing:

```json
{
  "rewrites": [
    { "source": "/(.*)", "destination": "/index.html" }
  ]
}
```

### 3.6. Opcao B — Deploy no Netlify (gratis)

Criar arquivo `netlify.toml` na raiz:

```toml
[build]
  command = "npm run build"
  publish = "dist"

[[redirects]]
  from = "/*"
  to = "/index.html"
  status = 200
```

### 3.6. Opcao C — Deploy em VPS com Nginx

```bash
# Copiar build para o servidor
scp -r dist/* usuario@seu-servidor:/var/www/carretao/
```

Configuracao Nginx:

```nginx
server {
    listen 80;
    server_name carretaoautopecas.com.br www.carretaoautopecas.com.br;

    root /var/www/carretao;
    index index.html;

    # SPA — redireciona todas rotas para index.html
    location / {
        try_files $uri $uri/ /index.html;
    }

    # Cache de assets estaticos (JS, CSS, imagens)
    location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff2?)$ {
        expires 1y;
        add_header Cache-Control "public, immutable";
    }

    # Gzip
    gzip on;
    gzip_types text/plain text/css application/json application/javascript text/xml;
    gzip_min_length 256;

    # Security headers
    add_header X-Frame-Options "SAMEORIGIN" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;
}
```

Para HTTPS (obrigatorio em producao):

```bash
# Instalar Certbot (Let's Encrypt)
sudo apt install certbot python3-certbot-nginx
sudo certbot --nginx -d carretaoautopecas.com.br -d www.carretaoautopecas.com.br
```

## 4. Deploy do Backend (Supabase Edge Functions)

### 4.1. Login e link do projeto

```bash
# Login no Supabase
supabase login

# Linkar ao projeto existente
supabase link --project-ref aztdgagxvrlylszieujs
```

### 4.2. Estrutura de pastas esperada pelo CLI

O Supabase CLI espera a estrutura:
```
supabase/
  functions/
    server/          # Nome da funcao = "server"
      index.tsx      # Entrypoint da Edge Function
      kv_store.tsx
      seed.tsx
      validation.ts
      test-shipping-handler.ts
```

O CLI vai detectar automaticamente `server` como o nome da funcao.

### 4.3. Deploy da Edge Function

```bash
# Deploy da funcao "server"
supabase functions deploy server --no-verify-jwt

# IMPORTANTE: --no-verify-jwt porque a funcao faz verificacao manual
# de tokens via getAuthUserId() e isAdminUser()
```

> **Nota**: O deploy pode levar 30-60 segundos pelo tamanho do arquivo (~23k linhas).
> Isso e normal.

### 4.4. Configurar variaveis de ambiente (Secrets)

No dashboard do Supabase → Settings → Edge Functions → Secrets, ou via CLI:

```bash
# Obrigatorias (ja existem se o projeto e o mesmo)
supabase secrets set SUPABASE_URL="https://aztdgagxvrlylszieujs.supabase.co"
supabase secrets set SUPABASE_ANON_KEY="sua-anon-key"
supabase secrets set SUPABASE_SERVICE_ROLE_KEY="sua-service-role-key"
supabase secrets set SUPABASE_DB_URL="sua-db-url"

# reCAPTCHA
supabase secrets set RECAPTCHA_SECRET_KEY="sua-chave-recaptcha"
supabase secrets set RECAPTCHA_SITE_KEY="sua-site-key-recaptcha"

# SafraPay (bloqueada aguardando homologacao — manter mesmo assim)
supabase secrets set SAFRAPAY_MERCHANT_ID="seu-merchant-id"
supabase secrets set SAFRAPAY_MERCHANT_TOKEN="seu-merchant-token"

# SMTP (quando configurar)
# supabase secrets set SMTP_HOST="smtp.seuservidor.com"
# supabase secrets set SMTP_PORT="587"
# supabase secrets set SMTP_USER="email@seudominio.com"
# supabase secrets set SMTP_PASS="sua-senha"
```

### 4.5. Verificar deploy

```bash
# Testar o endpoint de health
curl https://aztdgagxvrlylszieujs.supabase.co/functions/v1/make-server-b7b07654/health \
  -H "Authorization: Bearer SUA_ANON_KEY"

# Resposta esperada: {"status":"ok"} ou similar
```

## 5. Variaveis de Ambiente — Referencia Completa

| Variavel | Onde | Obrigatoria | Descricao |
|---|---|---|---|
| `SUPABASE_URL` | Edge Function | Sim | URL do projeto Supabase |
| `SUPABASE_ANON_KEY` | Edge Function + Frontend | Sim | Chave publica anon |
| `SUPABASE_SERVICE_ROLE_KEY` | Edge Function APENAS | Sim | Chave admin (NUNCA no frontend!) |
| `SUPABASE_DB_URL` | Edge Function | Sim | URL direta do PostgreSQL |
| `RECAPTCHA_SECRET_KEY` | Edge Function | Sim | Validacao server-side do reCAPTCHA |
| `RECAPTCHA_SITE_KEY` | Edge Function | Sim | Chave publica do reCAPTCHA |
| `SAFRAPAY_MERCHANT_ID` | Edge Function | Nao* | ID do lojista SafraPay |
| `SAFRAPAY_MERCHANT_TOKEN` | Edge Function | Nao* | Token SafraPay |
| `SMTP_HOST` | Edge Function | Nao** | Host do servidor SMTP |
| `SMTP_PORT` | Edge Function | Nao** | Porta SMTP (587 ou 465) |
| `SMTP_USER` | Edge Function | Nao** | Usuario SMTP |
| `SMTP_PASS` | Edge Function | Nao** | Senha SMTP |

\* SafraPay esta bloqueada aguardando homologacao
\** SMTP necessario para emails transacionais (confirmacao de pedido, etc.)

> **SEGURANCA**: A `SUPABASE_SERVICE_ROLE_KEY` NUNCA deve aparecer no frontend.
> Ela so existe no backend (Edge Function). Vazamento dessa chave compromete
> todo o banco de dados.

## 6. Configuracao do Banco de Dados

O banco de dados **ja esta configurado** no Supabase. Nao precisa rodar migrations.

O sistema usa uma unica tabela `kv_store_b7b07654` (key-value) que ja existe.
Todos os dados (produtos, pedidos, configuracoes, etc.) sao armazenados como
JSON nessa tabela via o utilitario `kv_store.tsx`.

Se voce precisar verificar:

```sql
-- No Supabase SQL Editor:
SELECT COUNT(*) FROM kv_store_b7b07654;
SELECT key FROM kv_store_b7b07654 LIMIT 20;
```

## 7. DNS e Dominio

### Se usar Vercel/Netlify:

1. Adicione o dominio customizado no dashboard da plataforma
2. Configure os registros DNS:
   - `A` record → IP fornecido pela plataforma
   - `CNAME` record → `cname.vercel-dns.com` (Vercel) ou equivalente

### Se usar VPS:

1. Aponte o dominio para o IP do servidor via registros `A`
2. Configure o Nginx com o `server_name` correto
3. Instale certificado SSL via Certbot (Let's Encrypt)

## 8. CORS — Atualizar Origem Permitida

Atualmente o backend aceita todas as origens (CORS aberto).
Para producao, voce pode restringir no `index.tsx` do servidor:

```typescript
// No index.tsx, o middleware CORS ja existe.
// Para restringir, altere a config para:
app.use("*", cors({
  origin: ["https://carretaoautopecas.com.br", "https://www.carretaoautopecas.com.br"],
  allowMethods: ["GET", "POST", "PUT", "DELETE", "OPTIONS"],
  allowHeaders: ["Content-Type", "Authorization", "X-User-Token"],
}));
```

## 11. Problemas Comuns

### "Failed to fetch" ou "Network error" ao carregar produtos
- **Causa**: Edge Function fria (cold start de 5-10s)
- **Solucao**: O warmup automatico em `api.ts` ja lida com isso. Se persistir, verifique se a Edge Function foi deployada corretamente.

### Pagina em branco ao navegar
- **Causa**: Servidor nao redireciona rotas para `index.html`
- **Solucao**: Configurar rewrite/redirect no servidor (veja secao 3.6)

### "Unauthorized" em rotas admin
- **Causa**: Token de autenticacao expirado ou usuario nao esta na whitelist
- **Solucao**: Verificar se o email admin esta correto e se a `SUPABASE_SERVICE_ROLE_KEY` esta configurada

### Build falha com erro de TypeScript
- **Causa**: Algumas dependencias podem precisar de versoes especificas
- **Solucao**: Usar `npm install` com as versoes exatas do `package.json`

### CORS error no browser
- **Causa**: Dominio do frontend nao permitido no backend
- **Solucao**: Verificar middleware CORS no `index.tsx` (secao 8)
